# Project Guidelines

## Purpose
このリポジトリは日々の学習内容を記録するための学習ログ集である。
新しいログを作るときは、既存の記法と粒度に合わせる。
参考スタイル:
- Swift系の学習ログは logs/swift 配下の既存ファイル
- Agentic Coding系の学習ログは logs/agentic-coding 配下の既存ファイル

## Log Structure
日次ログは原則として以下の構成にする。
- タイトルは日付
- 今日の作業内容
- 今日学んだこと
- 必要なら実装したこと
- 必要なら明日やること

## Writing Style
- 学習内容は箇条書きだけで終わらせず、何が分かったかを短く説明する
- 1つの見出しに複数の学びを詰め込みすぎず、学びは論点ごとに細かく分ける
- 「何が分かったか」だけでなく、「なぜそうなるか」「どこが重要か」「何を誤解していたか」まで書けるときは書く
- 単なる作業記録ではなく、再読したときに理解が戻る粒度でまとめる
- コードや表は必要なときだけ使う
- 既存ログの見出しレベルと雰囲気をなるべく揃える
- 日本語で記述する

## Validation
- 元情報をそのまま転記せず、用語の誤り、言い間違い、前後の矛盾があれば意味が通る形に補正する
- 学びとしてまとめる前に、元情報の中で内容が食い違っていないか確認する
- 明らかに誤っている説明は残さず、正しい理解に直してから書く
- 補正に確信が持てない場合は断定せず、不足点や確認事項として残す

## Classification
- Swift学習は logs/swift/YYYY-MM-DD.md
- Agentic Coding学習は logs/agentic-coding/YYYY-MM-DD.md
- どちらにも当てはまらない場合は、ユーザーに確認するか、最も近いカテゴリを提案する

## README Update
新しいログを追加したら README の該当セクションにも 1 行追加する。
表示形式は既存の箇条書きと揃える。

## Commit Workflow
- 日次ログを新規作成または更新したら、作成後に差分を確認する
- 差分確認では、今回のログ作成に関連するファイルだけが変更対象かを確認する
- 関連ファイルは原則として、追加または更新したログファイルと README に限る
- 変更内容の確認後は、その内容に合う短いコミットメッセージを作成する
- 学習ログ作成の依頼では、明示的に不要と言われない限り、コミットメッセージ案の作成まで行う
- コミットまで求められている場合は、関連ファイルだけをステージしてコミットまで進める
- 作業ツリーに無関係な変更がある場合でも、それらはステージせず、今回のログ作成に関係するファイルだけでコミットする

## Input Handling
入力が会話ログ、断片メモ、FigJam要約であっても、内容を整理して日次ログ形式に再構成する。
学びは大項目だけで済ませず、元情報から拾える論点を細かく分解して整理する。
会話中のラフな表現や一時的な誤解は、そのまま残すのではなく、最終的な理解として自然な文章に直す。
情報が足りないときは勝手に断定せず、曖昧な点を残さないよう補完候補を明示する。

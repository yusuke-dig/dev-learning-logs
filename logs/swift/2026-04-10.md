## 今日学んだこと

### `FetchDescriptor` の条件指定

```swift
// 全件取得（条件なし）
context.fetch(FetchDescriptor<Moment>())

// 条件あり（ロック済みのバッジのみ）
context.fetch(FetchDescriptor<Badge>(predicate: #Predicate { $0.timestamp == nil }))
```

条件を何も指定しなければ全件取得がデフォルト。KotlinのJPAの `findAll()` と同じ概念。

---

### `switch` の `where` による追加条件

```swift
switch badge.details {
case .firstEntry where moments.count >= 1:
//   ^^^^^^^^^^^ ^^^^^ ^^^^^^^^^^^^^^^^^^
//   caseの一致   かつ  追加条件
    newlyUnlocked.append(badge)
default:
    continue
}
```

`where` を使うことで「caseの一致」と「追加条件」を1行で書ける。`where` がない場合は `if` を中に書く必要があり冗長になる。

`default: continue` の意味：
- `break` → switch文を抜ける
- `continue` → forループの次の繰り返しへ進む

### バッジのロック・アンロックの状態管理

```swift
// ロック済み → timestamp が nil
// アンロック済み → timestamp に日時が入っている
```

```swift
// アンロック時の処理
badge.moment = newMoment          // どのMomentで解除されたか
badge.timestamp = newMoment.timestamp  // いつ解除されたか
```

PostgreSQLで例えると：
```sql
-- アンロック時
UPDATE badges
SET timestamp = '2026-04-10 12:00:00',
    moment_id = 'xxxx-xxxx'
WHERE id = ?;
```

### サンプルデータでのバッジ解除の再現

```swift
// ループの中で毎回判定することで段階的なバッジ解除を再現
for moment in Moment.sampleData {
    context.insert(moment)
    try badgeManager.unlockBadges(newMoment: moment)  // 毎回判定
}
```

ループの外で呼ぶと最後の1件しか判定できず、途中のバッジ解除が再現できない。

### `loadSampleMoments` はプレビュー専用

```swift
init(includeSampleMoments: Bool = false) {
//                                 ^^^^^デフォルトはfalse
    if includeSampleMoments {
        try loadSampleMoments()  // trueのときだけ実行
    }
}
```

| 環境 | `includeSampleMoments` | `loadSampleMoments` |
|---|---|---|
| `#Preview` / `.sampleDataContainer()` | `true` | 実行される |
| シミュレーター・実機 | `false`（デフォルト） | 実行されない |

シミュレーターでは `DataContainer()` のデフォルトが使われるため、ユーザーが自分でMomentを追加する本番の動作になる。

## 今日学んだこと

### SwiftData関連

- **`loadTransferable`** とは、`PhotosPickerItem` のメソッドで、選択された写真データを非同期で読み込むもの。`type` パラメータで `Data.self` や `UIImage.self` など取得する型を指定する。iCloudやディスクI/Oが発生するため `async/await` で処理する。

- **`context.insert` と `context.save` の違い**
    - `context.insert` → メモリ上のcontextに追加するだけ（まだSQLiteには書かれていない）
    - `context.save` → ディスク（SQLite）に書き込んで永続化する
    - SwiftDataの内部ストレージはSQLiteであり、Core DataをラップしたAPIになっている。

- **`@Query`** は SwiftDataから自動でデータを取得し、変化があればUIを自動更新するマクロ。`context.save()` したら自動でリストに反映される。

### SwiftUI関連

- **`loadTransferable` の `if let` が2つある理由**  
  `imageData: Data?` のOptionalアンラップ と `UIImage(data:)` の失敗可能イニシャライザのアンラップの2段階が必要なため。

- **`flatMap` の使い方**（SwiftのflatMapは2種類ある）
    - Optionalに対する `flatMap` → `Optional<Optional<T>>` を `Optional<T>` に平坦化する
    - 配列に対する `flatMap` → TypeScriptと同じく配列の入れ子を平坦化する
    - 本質は「コンテナの入れ子を一段つぶしながら変換する」こと

- **`@Observable` と `ObservableObject` の違い**（新旧の違い）

  | | `ObservableObject`（旧・iOS13〜） | `@Observable`（新・iOS17〜） |
    |---|---|---|
  | プロパティ | `@Published` を個別に付ける | 何も付けなくてOK |
  | 渡し方 | `.environmentObject()` | `.environment()` |
  | 受け取り方 | `@EnvironmentObject` | `@Environment` |
  | 監視精度 | クラス単位 | プロパティ単位（無駄な再描画が少ない） |

- **`@MainActor`** とは「このクラスの処理をすべてメインスレッドで実行する」宣言。`mainContext` はメインスレッド専用APIのため必須。メインスレッドに集中させることで、複数スレッドからの同時アクセスによるRace Conditionを防ぐ。

- **メインスレッドとバックグラウンドスレッドの違い**
    - メインスレッド：アプリ起動時にOSが自動で1本だけ作る特別なスレッド。UI描画・タッチイベント処理を担う。
    - バックグラウンドスレッド：必要なときに作って終わったら捨てる。複数存在できる。

- **`@Environment(\.dismiss)` と `@Environment(DataContainer.self)` の違い**
    - `\.` から始まるもの → Apple公式の `EnvironmentValues` に定義済み（`\.dismiss`、`\.colorScheme` など）
    - 型を直接渡すもの → 自分で定義した `@Observable` なクラス

- **ジェネリクス `<Content: View>`** → 「ContentはViewに準拠した何らかの型」という意味。VStackやHStackも内部的に同じ書き方。TypeScriptのジェネリクスと同じ概念。

- **`ContentUnavailableView`**（iOS17〜）→ データが空のときに表示する専用のUI部品。アイコン・タイトル・説明文をシンプルに表示できる。

- **`Text(timestamp, style: .date)`** → 日付のフォーマットを簡単に指定できる。ロケール（地域・言語設定）に応じて自動でフォーマットが変わる。

- **`Color.clear` をZStackに入れる理由**  
  `Color` は親View いっぱいに広がる性質がある。`ZStack(alignment: .bottom)` で `Color.clear` を入れることでZStackを親のサイズまで広げ、`alignment: .bottom` を正しく機能させるためのサイズを確保できる。

- **`.confirmationDialog`** → ボタンを押したときにダイアログを表示するモディファイア。`isPresented` のフラグがtrueになると表示される。`role: .destructive` でボタンを赤色にして危険な操作であることを示す。

- **`.fixedSize(horizontal:vertical:)`** → Viewが圧縮・引き伸ばされるのを防いで本来のサイズを維持するモディファイア。`frame(maxHeight:)` で制限をかけつつ、テキストが潰れないようにするために使う。

### extension Viewについて

```swift
extension View {
    func sampleDataContainer() -> some View {
        self
            .environment(sampleContainer)
            .modelContainer(sampleContainer.modelContainer)
    }
}
```

KotlinやSwiftの拡張関数と同じ概念。`extension View` にメソッドを定義すると、すべてのViewで `.` で呼び出せるようになる。Apple公式のモディファイア（`.padding()` など）も内部的に同じ仕組み。

### ModelContainer と ModelContext の役割

```
ModelContainer（DB本体・SQLiteへの接続）
　↓ から生成
ModelContext（操作口）
　├── insert()   書き込み
　├── delete()   削除
　└── save()     ディスクへの永続化

ModelContainer
　↓ を直接参照
@Query（読み取り専用マクロ）
```

`.environment(dataContainer)` と `.modelContainer(dataContainer.modelContainer)` を両方渡す必要があるのは、`@Query` が `DataContainer` ではなく `ModelContainer` を直接参照しているため。

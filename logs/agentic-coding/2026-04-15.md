## 今日学んだこと

### Agentの役割設計

今回初めて `.github/agents/` に agent ファイルを作成した。
各 agent に明確な責務を持たせることで、TDD の Red-Green サイクルを AI に任せられる。

作成した agent と役割は以下のとおり。

| agent | 役割 |
|---|---|
| `orchestralator` | 全体進行の指揮者。自分では実装せず、適切な順序で他 agent へ委譲する |
| `pm` | 要求を整理し、TDD スライスへ分解する。コードは書かない |
| `tester` | 最小の失敗テスト（Red）を書き、Red であることを確認してから実装者へ渡す |
| `implementer` | 失敗テストを通す最小限の本番コードだけを書く（Green）。拡張はしない |
| `e2e` | 全 TDD スライス完了後にユーザーフロー視点の E2E テストを実行する |
| `security` | 最後にセキュリティ観点でレビューし、リスクを報告する |

#### 実行順序（orchestralator が管理）

```
PM → tester(Red) → implementer(Green) → tester(検証) → ...（スライス繰り返し）
                                                          → e2e → security
```

各 agent の出力形式（`Current phase:` / `Next handoff:` など）を統一したことで、
orchestralator が次の委譲先を判断しやすくなっている。

### Hooksの設定

`.github/hooks/hooks.json` に以下の7イベントをすべて登録した。

```json
SessionStart / UserPromptSubmit / PreToolUse / PostToolUse
SubagentStart / SubagentEnd / SessionEnd
```

各イベントで `log-execution.sh` を呼び出し、`.github/hooks/logs/execution.jsonl` に JSONL 形式で記録する。

#### ログスクリプトのポイント

```bash
# stdin から渡ってくる JSON を読む
INPUT=$(cat)

# snake_case / camelCase 両方に対応するフィールド取り出し関数
field() {
  for key in "$@"; do
    val=$(echo "$INPUT" | jq -r ".$key // empty" 2>/dev/null)
    [ -n "$val" ] && echo "$val" && return
  done
}
```

イベントによって渡ってくるフィールド名が変わる（例: `tool_name` vs `toolName`）ため、
複数のキー名を試してフォールバックする helper 関数を用意した。

### Skillsの設定

`.github/skills/test-execute/SKILL.md` にテスト実行スキルを定義した。

```markdown
frontendのディレクトリに移動して `npm run test` を実行する。
backendのディレクトリに移動して `./gradlew test` を実行する。
```

「テストを実行」と言うだけでフロント・バックエンド両方のテストを走らせられる。
tester / e2e agent がこのスキルを呼び出すことでテスト実行の手順を外部化できた。

#### `name` と `description` は特に重要

SKILL.md の `name` と `description` をしっかり書いておかないと、agent がそのスキルを選んで読んでくれない。
**いつ使われるスキルなのかを明記する**ことが呼び出し精度に直結する。

#### コンテキスト節約のためにSKILLへ切り出す

agent ファイルにすべての手順を書くとコンテキストが肥大化する。
繰り返し使う処理は SKILL に切り出して省エネ設計にする。

### Agentアーキテクチャの設計原則

#### メインagentはシンプルに保つ

オーケストレータは「委譲して結果をもらうだけ」の設計にすると上手く動く。
具体的な処理と結果の責任はサブエージェントに持たせる。

```
orchestralator → PM・tester・implementer・e2e・security へ委譲
              ← 結果だけ受け取る
```

自分で実装・テストを抱え込むと、判断とコンテキストが混濁して挙動が安定しない。

#### サブエージェントに `tools` を明示する理由

agent の `tools` を省略すると、メインエージェントの tools がそのまま引き継がれてしまい思わぬ動作をする。
サブエージェントに与えるべきツールだけを明示することで、権限の境界をコントロールできる。

```yaml
# tester: テストコードの編集と実行ができる。本番コードは編集しない
tools: [read, search, edit, execute]

# pm: 編集はできるが実行はしない
tools: [read, search, edit]
```

#### 申し送りファイルでオーケストレータを薄くする

オーケストレータの中に実行順序をすべて書くと肥大化する。
各 agent への申し送り内容をファイルとして外部化すると、オーケストレータが「呼ぶだけ」の薄い部品に保てる。

### ハーネスとは

ハーネスとは**エージェントを動かす実行基盤そのもの**のこと。元はソフトウェアテストの「テストハーネス」から来た用語。

```
┌─────────────────────────────────────┐
│            ハーネス                  │  ← GitHub Copilot の実行エンジン本体
│  ・ツール実行の管理                    │
│  ・ライフサイクルイベントの発火          │
│  ・エージェントの起動/終了              │
│  ・コンテキストの保持                  │
├─────────────────────────────────────┤
│  agents / hooks / skills            │  ← 今回作ったもの（ハーネスへの設定）
└─────────────────────────────────────┘
```

今回作った `.github/agents` `.github/hooks` `.github/skills` はハーネス自体ではなく、**ハーネスにどう動いてほしいかを定義したもの**。ハーネスを活用した設計であって、ハーネスを作ったわけではない。

hooks が「100% 実行される」のも、hooks がハーネスのライフサイクルに組み込まれているから。

---

### Hooksの特性

Hooks は**100% 実行される決定論的なゲート**。
agents や skills は「お願いベース」なので実行されないこともあるが、hooks だけは必ず発火する。

```
hooks   → 決定論的。必ず動く。検知して副作用を起こすもの。
agents  → お願いベース。指示通り動かないこともある。
skills  → お願いベース。description が曖昧だと呼ばれない。
```

この特性を活かして、「絶対に記録したい」「絶対に止めたい」処理は hooks に置く。

### バイブコーディングとの比較

マルチエージェントで TDD サイクルを回す方式は、バイブコーディング（1つの AI に全部お願いする）と比べて**処理が遅くなる**。
agent 間のハンドオフや hooks の発火がオーバーヘッドになるため。
品質・制御性と引き換えにスピードを失うトレードオフとして認識しておく。

複数プロジェクトが入れ子になっている場合、対象プロジェクトのルートで `git init` してからでないと
GitHub Copilot がうまくファイルを読み込めないことがある。

### 高品質なプロダクトを作るテクニック

#### Plan → Generate → Evaluate ループ

デザイン品質を上げたいとき・リファクタリングさせるときは、このループを繰り返させる。

```
Plan（設計） → Generate（生成） → Evaluate（評価） → Plan（再設計）...
```

Evaluate が特に重要で、**強化学習的に批判的・厳しめな Evaluator Agent** を用意すると品質が上がる。

#### 自己学習ループ（レトロスペクティブ）

実装後に AI 自身に振り返り（レトロ）させることで、同じ間違いを繰り返さないようにできる。

#### セキュリティはOWASPなど具体的な基準を明記する

「セキュリティを確認して」という曖昧な指示より、OWASP Top 10 など具体的な観点を書いた方が精度が上がる。

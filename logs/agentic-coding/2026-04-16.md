## 今日学んだこと

### 既存の良いプラグインを再実装すると仕組みが見えやすい

最初から完全にゼロベースで考えるより、既存の良いプラグインの振る舞いを再実装してみる方が学習しやすい。
外から見える挙動を手掛かりに組み立てると、どの設定ファイルや責務分離が本質なのかを理解しやすい。

また、AI に丸投げするよりも補助として使い、自分の手で試行錯誤した方が原則の理解が深まる。

---

### Agent と Skill は最小構成から始める

Skill から作り始めても Agent から作り始めてもよく、固定の順序はない。
重要なのは、最初から要素を増やしすぎないことだった。

まずは 1 つの Agent で成り立つかを考え、成立しない理由が見えてから分割する方が整理しやすい。
Skill も最初から大量に作るのではなく、Agent 間で重複する手順や繰り返し使う処理が見えてから切り出すのが自然だと分かった。

前日に試した「先に Agent を多く作ってから Skill に分ける」形は、ハーネスを体験するための例であって、常にその流れが最適というわけではない。

---

### プラグインは提供物の入口を明示して配布する

プラグインでは `marketplace.json` に登録し、専用ディレクトリと `plugin.json` を用意して、そのプラグインが何を提供するのかを明示する。
Skills・Agents・Hooks のような複数の構成要素をまとめて配布できるため、設定を都度コピーするより再利用しやすい。

README に使い方や補足を残しておくことも重要で、導入後に「何のためのプラグインか」を追いやすくなる。

運用方法はツールごとに差があり、Copilot ではグローバルにインストールされる前提で扱うケースがあると分かった。
そのため、不要になったプラグインをどう管理するかまで含めて考える必要がある。

---

### Hooks は必ず通る制御点として使える

Hooks の強みは、Agent の判断に依存せずイベント時に確実に実行できる点にある。
特に `PreToolUse` は権限が強く、ツール実行前に介入して「このタイミングでは使わせない」といった制御ができる。

一方で、prompt や Agent 側の条件分岐に寄せすぎると AI 判断に依存しやすくなり、「必ず実行される」という Hooks の強みが薄れることも理解できた。

また、Hooks のロガーを見直した結果、CLI 側ではサブエージェント開始ログが取得できず、Chat では取得できる挙動差があった。
実装差かバグかはまだ断定できないが、少なくとも観測方法によって取れるログが違うことは押さえておく必要がある。

---

### オーケストレーションは用途ごとに型を選ぶ

オーケストレーションには並列・直列・協調などいくつかの型があるが、迷ったときは hub-and-spoke が汎用的だと学んだ。
独立した調査やリファクタのように、各作業を疎結合で進めたい場面と相性が良い。

一方、relay/chain は枝分かれや戻りが少ない定型手順に向いている。
agent team は大規模なフルスタック開発のような場面で有効だが、同時起動コストが高く、エージェント間の引き継ぎ漏れも起こりやすい。

hub-and-spoke を成立させるには、中心に返すコンテキストの粒度を意識して設計することが重要だと理解した。

---

### 手を動かして試すこと自体が学習になる

「人気のプラグインを自分たちで作ってみる」という課題の価値は、便利機能を増やすことそのものよりも、責務分離や制御点を自分で理解するところにある。

例えば、CI で長すぎるテスト名を検出して修正候補まで出すプラグインのように、具体的な困りごとから作ると、何を自動化すべきかが見えやすい。
自分で文章や指示を変えながら試行錯誤することが、自己学習すべきポイントだと分かった。

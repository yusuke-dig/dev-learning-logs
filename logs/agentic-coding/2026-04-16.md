## 今日学んだこと

### プラグインはゼロから全部作るより、良い仕組みを再実装して理解する

最初から完全に独自の仕組みを作るより、便利だと思うプラグインの構造を自分たちで再実装してみる方が学びが深い。
特に、お手本の中身を先に見ずに試すと、どこが本質的な設計要素なのかを自分の頭で切り分けやすい。

---

### Agent と Skill は最初から増やしすぎない

SKILL から作るか Agent から作るかに絶対的な順序はないが、最初から細かく分けすぎる必要はないと分かった。
まずは必要最低限の Agent で成立するかを見て、重複する処理が見えてきたら Skill に切り出す方が自然。

「まず大きなドキュメントを書いてから構成を分ける」やり方もあるが、最初から多エージェント前提で設計すること自体が必須ではない。

---

### プラグインは Skill・Agent・Hooks をまとめて配布できる

プラグイン作成では、ディレクトリを作り、その中に plugin.json を置き、必要に応じて marketplace 側へ登録する流れになる。
プラグインは Skill、Agent、Hooks などをまとめて提供できるので、毎回手作業で設定を配るより再利用しやすい。

一方で、Copilot 側はプラグインがグローバルインストール前提なので、不要になったらアンインストールが必要になる。
Claude はプロジェクト単位で扱えるという違いがあり、適用範囲の設計も意識する必要がある。

---

### Hooks は決定論的に動かしたい処理に向いている

Hooks の中でも PreToolUse は権限が強く、ツール実行前に介入してキャンセルできる。
そのため、「この条件ではこのツールを使わせない」のような制御を AI の判断に委ねずに入れられる。

prompt や agent の指示に書くだけだと、最終的には AI の判断になる。
必ず実行したい処理や必ず止めたい処理は、指示文ではなく Hooks に置く方が安定する。

---

### オーケストレーションは迷ったら hub-and-spoke を起点に考える

オーケストレーションには、hub-and-spoke、リレー/チェーン、エージェントチームのような型がある。
汎用的で扱いやすいのは hub-and-spoke で、独立した調査やリファクタのような作業に向いている。

リレー/チェーンは、手順がはっきり決まっていて分岐戻りが少ない処理に向いている。
一方で、エージェントチームは大規模な並列開発に向くが、同時起動コストやコンテキスト引き継ぎ漏れのリスクが大きい。

---

### Chat と CLI では観測できる情報が違うことがある

Hooks のロガーを見たとき、CLI ではサブエージェント開始が取れず、Chat では取れるという差分があった。
バグの可能性もあるが、少なくとも「どこで何が観測できるか」は実行環境で変わることがある。

そのため、ログ取得や検証フローを設計するときは、機能仕様だけでなく実行環境ごとの差も確認する必要がある。
